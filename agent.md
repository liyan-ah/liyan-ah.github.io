# 博主行文风格指南

> 本文件总结了历史博客的行文风格模式，用于指导后续博客写作，确保风格一致性。
> 基于对 content/ 下 30+ 篇技术博客的分析整理，最后更新于 2026-04-29。

---

## 一、总体风格定位

**"工程师的学习笔记"** —— 以第一人称视角，记录工作中遇到的技术问题及其探索过程。
务实导向、口语化表达、谦逊态度。不是教科书，而是"笔者搞懂后的整理分享"。

## 二、Front Matter 规范

### 2.1 标准模板（TOML 格式）

```toml
+++
title = "文章标题"
description = "一句话描述文章内容概要，20~50字"
date = "2026-04-28T20:10:58+08:00"

[taxonomies]
categories = ["分类名"]
tags = ["标签1", "标签2", "标签3"]
+++
```

### 2.2 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `title` | ✅ | 简洁明了，可含英文技术术语 |
| `date` | ✅ | ISO 8601 格式，带引号或不带均可 |
| `description` | 推荐 | 文章摘要，帮助 SEO 和读者快速了解 |
| `categories` | ✅ | 见下方分类体系 |
| `tags` | ✅ | 具体技术点，2~5 个 |
| `comments` | 可选 | `true` 表示希望读者讨论 |
| `draft` | 可选 | `true` 表示未完成草稿 |
| `author` | 可选 | 早期文章使用 `"李岩"`，新文章可省略 |

### 2.3 分类体系（Categories）

| 分类 | 适用场景 |
|------|----------|
| `BPF` | eBPF/BPF 相关技术实践 |
| `程序人生` | 工具配置、环境搭建、经验分享、译文 |
| `读书` | 读书笔记、人文历史类 |
| `技术栈` | 技术体系学习系列（如 datafusion-learning） |
| `理论内容` | 协议、算法、原理分析 |
| `代码之美` | 设计模式、优雅实现 |

## 三、文章结构规范

### 3.1 技术文章标准结构（最常用）

```
> 背景说明 / 写作动机（引用块形式，1~3句话）
> 通常以"最近工作中..."、"开始搞xx了..."、"算起来距离上一次..."开头

<!--more-->                          ← 可选，首页摘要截断点

# 一、效果展示 / 问题现象
  先看效果/先说问题，吸引读者（"按照惯例，我们还是来看下效果"）

# 二、原理分析 / 方案设计
  引用外部文档（RFC/官方文档/他人文章），结合自己的理解阐述

# 三、实现细节 / 代码展示
  完整可运行的代码，配合执行结果
  代码中用注释标注：文件路径、执行阶段、关键注意点

# 四、风险 / 注意事项 / 待确认 / 优缺点
  诚实列出不足之处和未解决的问题（"笔者的困惑"、"待确认"）

以上，周末愉快～                       ← 或类似结尾
```

### 3.2 理论综述型结构（如 storage-introduction）

```
> 学习动机（引用块）
> "最近工作中一直接触xx，感觉了解的还少..."

## 简介
  总起：领域概述 + 本文目标 + 内容来源声明

## 主题A（如 MySQL）
  本节期望能够回答以下问题：
  * 问题1？
  * 问题2？

### A.1 简述
### A.2 核心机制
### A.3 待确认          ← 诚实标记未完成部分

## 主题B（如 ES）
  ... 同上结构 ...

## 后记
  感慨篇幅所限、资料缺失、后续计划
  "本次且这样吧。"

[^1]: url   ← 大量脚注引用
```

### 3.3 学习系列型结构（如 datafusion-learning）

```
> 系列背景（引用块）
> "开始搞xx了，项目中用到的很多基础知识还是欠缺的..."

## 大主题一
> 子说明（嵌套引用块，概括本节内容）

## 大主题二
> 子说明

...
（无固定结尾，系列持续更新）
```

### 3.4 译文型结构

```
> 译者注：原文由笔者翻译并提交至 [链接]，同步转载到此处。

> "名言引用..."    ← 如原文有名言

正文翻译内容...

原文[点这里]查看

<!-- 参考链接 -->
[标识]: url       ← 引用式链接，文末集中定义
```

## 四、语言风格细则

### 4.1 人称与自称

- **统一使用"笔者"**作为第一人称自称，不用"我"（极少数除外）
- 偶尔用"笔者理解"、"笔者感觉"、"笔者试着"等谦逊表达
- 不使用"你"、"您"等第二人称（非教程型文章）

### 4.2 中英文混用规则

- **技术术语保留英文**：`goroutine`, `channel`, `uprobe`, `B+树`, `Parquet`, `RowGroup`
- **常用缩写直接使用**：`BPF`, `TCP`, `ES`, `CK`, `IOx`, `API`, `RFC`, `OLAP`
- **首次出现的术语可中文解释**："ES，也是经典的存储了"，"CK 同样有诸多的表引擎"
- **工具/库名称保留英文**：`Clippy`, `bpftrace`, `InnoDB`, `MergeTree`
- **内联代码用反引号**：`` `chan-send` 的状态 ``，`` `tcp_close` 对服务主被调信息的观测 ``

### 4.3 口语化特征词汇

以下表达在博客中高频出现，属于风格标志：
- "搞"：`"开始搞存储了"`, `"再往下挖掘下"`, `"总得在23年结束前再搞两篇"`
- "折腾"：隐含在工具配置类文章的语境中
- "赏析"：用于译文系列
- "搜肠刮肚"：用于开篇感慨
- "算是...有始有终"
- "值的注意的是"（注意：常写为"值"而非"值得"）
- "从过往的经历中来看"
- "这里趁着这个机会，做一些总结"

### 4.4 标点符号习惯

- **正文使用中文标点**：逗号 `，` 句号 `。` 冒号 `：` 顿号 `、`
- **括号使用中文括号**：（）、但代码块内的注释可用英文括号
- **代码块内保持英文标点**
- **列表使用 `*` 或数字编号**

### 4.5 段落与长度

- **段落偏长**：一个段落经常包含 3~5 句话，构成完整的论述单元
- **不畏惧长句**：复合句、从句较多，一个句子可能包含多个分句
- **善于用逗号连接**：句子中间常有较多的解释性插入语

## 五、格式使用规范

### 5.1 引用块（Blockquote `>`）的使用场景

| 场景 | 示例 | 出现位置 |
|------|------|----------|
| 文章开篇背景 | `> 最近工作中一直接触存储相关的内容...` | Front Matter 后第一段 |
| 写作动机 | `> 开始搞存储了，项目中用到的很多基础知识还是欠缺的。` | 开篇 |
| 时间感慨 | `> 2023年就要结束了，算起来距离上一次更新也有很久了。` | 开篇 |
| 名言引用 | `> "看起来你在写信。需要什么帮助么？"` | 译文/技术文章开头 |
| 章节子说明 | `> Vibe Coding 时代，理解概念至关重要。` | 二级标题下方 |
| 原文摘录 | `> 很久以前，西方著名诗人西摩尼得斯...` | 读书笔记 |

### 5.2 脚注（Footnote `[^n]`）

- **格式**：`[^1]`, `[^2]`, ... 编号递增
- **位置**：文末集中列出
- **内容**：URL 链接为主，格式为 `[^n]: https://url。描述文字。`
- **使用场景**：理论综述型文章大量使用；技术实践型文章较少使用

### 5.3 代码块规范

**语言标注习惯：**

| 语言/场景 | 标注 |
|-----------|------|
| Rust | `` ```rust `` |
| Go | `` ```go `` 或无标注 |
| Python | `` ```python `` |
| C | `` ```c `` |
| bpftrace | 无标注或 `` ``` `` |
| 终端输出 | 无标注 |
| HTTP 报文 | `` ```http `` |

**代码注释风格（重要！）：**

```rust
// runtime/chan.go        ← 文件路径注释，标明源码出处
// terminal 1，启动服务   ← 终端命令说明，标注操作步骤
// 第一次，没有使用 ebpf 生效  ← 执行阶段/对比说明
// 请注意这里              ← 关键注意点提示
/* 本注释内容为译者添加 */     ← 译文专用译者注释
```

**代码展示原则：**
- 优先给出**完整可运行代码**
- 配合**执行结果展示**（终端输出原样粘贴）
- **对比场景**用注释区分（开启 vs 未开启、修改前 vs 修改后）

### 5.4 表格使用

- **对比场景**：存储系统对比、框架特性对比、方案优劣对比
- **格式**：标准 Markdown 表格，含表头分隔线
- **内容**：简洁扼要，不在表格中放大段文字

### 5.5 图片使用

- **语法**：`![描述](图片名.png)`
- **Alt 文本**：有时比较随意，如 `![overwrote existing file](xxx.png)` （可能是编辑器自动生成）
- **图片放置**：紧跟在引用它的段落之后

### 5.6 特殊标记

| 标记 | 含义 | 使用场景 |
|------|------|----------|
| `<!--more-->` | 首页摘要截断 | 开篇背景之后，正文开始前 |
| `<!-- 参考链接 -->` | 参考链接区块起始标记 | 译文型文章文末 |
| `draft = true` | 未完成草稿 | Front Matter 中 |
| `### x.x 待确认` | 未完成章节标记 | 正文中小节标题 |
| `<div class="ferris">` | Ferris螃蟹图标 | Rust 译文系列 |

## 六、章节命名习惯

### 6.1 一级标题（`#`）

Front Matter 之后通常直接使用一级标题作为正文开始：
- `# 一、tcp close 的一般过程`
- `# 一、追踪效果`
- `# 二、方案设计`
- `# 三、方案实现`
- `# 四、追踪的风险`
- `# Parquet 文件结构介绍`
- `# 在 Databse 中的使用`

特点：使用**中文数字编号**（一、二、三、），后跟简短的主题词。

### 6.2 二级标题（`##`）

- `## 简介`
- `## MySQL`
- `## Elasticsearch`
- `## ClickHouse`
- `## 后记`
- `## 核心概念介绍`
- `## 使用示例`
- `## 《翦商》介绍`

### 6.3 三级标题（`###`）

- `### 2.1 简述`
- `### 2.2 InnoDB`
- `### 2.3 待确认`
- `### 3.1 简述`
- `### 3.2 系统组成`
- `### 注释`

特点：使用**阿拉伯数字编号**（x.y 格式），后跟小节主题。

### 6.4 小节内的问题列表

在每个大主题开头，常用列表列出本节要回答的问题：

```
本节期望能够回答以下问题：
* MySQL 是否一定是 B+ 树的存储结构。
* InnoDB 为何选择 B+ 树数据结构。
* 一个 InnoDB 实例支持多少条数据记录。
```

## 七、结尾规范

### 7.1 常用结尾语

| 场景 | 结尾语 | 示例文章 |
|------|--------|----------|
| 周末发布的技术实践 | `以上，周末愉快～` | channel 追踪、gls 实现 |
| 节假日发布 | `以上，新春愉快～` | Hugo 迁移 |
| 一般技术文章 | `以上。` | ebpf 数据写入 |
| 总结性 | `以上，作为记录了部分总结。` | TCP close 分析 |
| 理论综述（未完） | `本次且这样吧。` | 存储概述 |
| 译文 | `原文[点这里]查看` + 参考链接区块 | Clippy 译文 |
| 读书笔记 | 无固定结尾，脚注结束 | 翦商、理想国 |

### 7.2 参考链接区块（译文专用）

```markdown
原文[点这里]查看

<!-- 参考链接 -->
[标识1]: https://url1
[标识2]: https://url2
```

## 八、Don'ts（避免事项）

1. **不要**使用 YAML 格式的 Front Matter（本项目使用 TOML，即 `+++` 分隔符）
2. **不要**在正文中过度使用一级标题（Front Matter 后一般直接开始正文的一级标题）
3. **不要**使用"你好"、"谢谢观看"等过于正式的开头/结尾
4. **不要**在代码块内使用中文标点（代码注释除外）
5. **不要**假装全知全能——诚实标记"待确认"、"笔者的困惑"、"笔者一时怕是无法有效的整理完成"
6. **不要**遗漏重要代码的**文件路径注释**（尤其是引用外部源码时）
7. **不要**使用"你"来称呼读者（这不是教程，是学习笔记）
8. **不要**过度使用形容词和修辞——保持工程师的客观朴实风格
